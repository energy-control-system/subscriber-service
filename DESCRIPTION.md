# Микросервис subscriber-service

## Общее описание

**subscriber-service** — микросервис, отвечающий за управление абонентами, объектами учета электроэнергии и договорами в системе учета электроэнергии. Микросервис реализован на языке Go и использует чистую архитектуру с разделением на слои: API, сервисы, репозитории.

### Основные функции микросервиса:

1. **Управление абонентами (Subscribers)** — создание, получение и хранение информации об абонентах, включая их паспортные данные
2. **Управление объектами (Objects)** — учет объектов потребления электроэнергии с привязанными приборами учета (devices) и пломбами (seals)
3. **Управление договорами (Contracts)** — создание и получение договоров, связывающих абонентов с объектами
4. **Парсинг реестра (Registry)** — импорт данных из Excel-файлов для массовой загрузки информации
5. **Обработка событий проверок (Inspections)** — подписка на события Kafka для автоматического обновления статуса абонента по результатам проверок

### Технологический стек:

- **Язык программирования**: Go
- **База данных**: PostgreSQL
- **Очередь сообщений**: Apache Kafka
- **HTTP-сервер**: собственная библиотека golib/gohttp
- **Миграции**: goose
- **Парсинг Excel**: excelize

## Описание диаграммы классов

Диаграмма классов (`class_diagram.puml`) отображает архитектуру микросервиса и взаимосвязи между компонентами.

### Основные компоненты диаграммы:

#### 1. Слой API (`api`)
- **ServerBuilder** — конструктор HTTP-сервера, отвечающий за регистрацию маршрутов и middleware
- **Handler-классы** — обработчики HTTP-запросов для каждой доменной области (Subscriber, Object, Contract, Registry)

#### 2. Слой сервисов (`service`)
Содержит бизнес-логику приложения, разделенную по доменам:
- **SubscriberService** — управление абонентами
- **ObjectService** — управление объектами и приборами учета
- **ContractService** — управление договорами и обработка событий проверок
- **RegistryService** — парсинг и импорт данных из Excel

Каждый сервис зависит от абстрактных интерфейсов репозиториев, что обеспечивает слабую связанность и возможность тестирования.

#### 3. Слой данных (`database`)
Реализации репозиториев для работы с PostgreSQL:
- **SubscriberDBRepository** — реализует интерфейсы работы с абонентами
- **ObjectDBRepository** — реализует интерфейсы работы с объектами
- **ContractDBRepository** — реализует интерфейсы работы с договорами

#### 4. Кластерный слой (`cluster`)
Компоненты для взаимодействия с другими микросервисами:
- **TaskClient** — HTTP-клиент для получения информации о задачах из task-service
- **Inspection** и **Event** — модели для обработки событий из Kafka

#### 5. Доменные сущности
- **Subscriber** — абонент с паспортными данными и статусом
- **Object** — объект учета с приборами (Device) и пломбами (Seal)
- **Contract** — договор, связывающий абонента с объектом
- **Task** — задача на проверку (из внешнего сервиса)
- **Inspection** — результат проверки

### Связи между компонентами:

1. **Композиция**: Subscriber содержит Passport; Object содержит Device; Device содержит Seal
2. **Агрегация**: Contract ссылается на Subscriber и Object
3. **Реализация интерфейсов**: Репозитории базы данных реализуют абстрактные интерфейсы сервисного слоя
4. **Зависимости**: Сервисы зависят от интерфейсов репозиториев (Dependency Injection)

### Применяемые паттерны:

- **Repository Pattern** — абстракция доступа к данным
- **Dependency Injection** — внедрение зависимостей через конструкторы
- **Builder Pattern** — построение HTTP-сервера через ServerBuilder
- **Event-Driven Architecture** — обработка событий проверок через Kafka

## Схема базы данных

Микросервис использует следующие таблицы PostgreSQL:

| Таблица | Описание |
|---------|----------|
| `subscribers` | Основная информация об абонентах |
| `subscriber_statuses` | Справочник статусов абонентов (Active, Violator, Archived) |
| `passports` | Паспортные данные абонентов |
| `objects` | Объекты учета электроэнергии |
| `devices` | Приборы учета (счетчики) |
| `device_place_types` | Справочник типов мест установки (Other, Flat, StairLanding) |
| `seals` | Пломбы на приборах учета |
| `contracts` | Договоры между абонентами и организацией |

## Взаимодействие с другими сервисами

1. **task-service** — получение информации о задачах на проверку через HTTP API
2. **inspection-service** — получение событий о завершенных проверках через Kafka (топик `inspections`)

При получении события о завершенной проверке типа "Верификация" или "Несанкционированное подключение" микросервис автоматически обновляет статус абонента на основании результатов проверки.
